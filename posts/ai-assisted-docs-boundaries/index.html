    AI 辅助技术文档的边界在哪里

    我第一次让 AI 补一段 API 响应示例时，它很快给出了一个结构完整的 JSON。

    字段名合理，状态码也像真的。问题在于，其中两个字段并不存在。这样的内容读起来很顺，也很容易在没有核对时混进文档。

    这件事让我开始认真区分：哪些工作可以交给 AI 加速，哪些判断必须留在作者手里。

    我会让 AI 处理什么

    AI 很适合处理已经有明确依据、但需要整理的内容。

    例如，我可以把接口定义、字段表和现有说明放在一起，让它协助检查：

    • 同一个术语是否出现了多种写法；
    • 请求参数和响应示例是否遗漏字段；
    • 一段说明是否包含过长的句子；
    • Quick Start 是否缺少前置条件、验证结果或下一步链接；
    • 标题层级和列表格式是否一致。

    这些工作有一个共同点：判断依据已经存在。AI 帮我扩大检查范围，最后仍然可以回到原始资料逐项确认。

    我也会用它比较两种结构方案。比如，一份 API 接入指南可以按“认证、请求、响应、错误处理”组织，也可以按“第一次成功调用”的操作顺序组织。AI 可以快速列出两种目录，我再根据读者任务做取舍。

    哪些判断必须由作者负责

    技术文档里最危险的错误往往很像正确答案。

    默认值、版本差异、错误码触发条件、寄存器位定义和命令输出，都需要可靠来源。只要我无法指出某句话来自哪份规格、哪个接口定义或哪次实际验证，它就不应该以确定语气出现在交付稿里。

    读者判断也无法外包。AI 不知道目标读者是否已经配置环境，不知道技术支持最常收到哪类问题，也不知道某个步骤失败后会产生多大成本。这些信息来自项目语境、访谈、测试和反馈。

    因此，我会把下面四项责任留给自己：

    1. 确认事实来源和适用版本。
    2. 决定页面服务哪类读者和任务。
    3. 运行命令、请求或操作步骤，验证结果。
    4. 对最终发布内容负责。

    我目前使用的工作流

    在这个作品集中，我把 AI 放在“整理”和“复核”两个位置。

    起稿前，我先准备来源材料，写清目标读者和完成标准。结构确定后，AI 可以协助整理术语、发现遗漏和提出另一种表达。完成初稿后，我再回到原始资料检查参数、链接、代码和页面路径。

    以IoT 接口集成指南为例，它使用的是虚构设备场景。即便如此，文档内部仍需自洽：认证流程要能连接到请求示例，参数表要和 URL 一致，响应字段要能对应错误处理。AI 可以帮我发现表面不一致，接口是否真正说得通仍需要我逐段检查。

    对于 Docs-as-Code 页面，我还会使用 Vale 和构建检查处理可重复的规则。AI、Vale 和人工检查承担不同任务：一个帮助整理语言，一个执行确定规则，一个负责技术判断。

    我的发布标准

    我现在会在发布前问自己三个问题：

    • 这条信息的来源在哪里？
    • 读者能否按文档完成一次操作？
    • 出现失败时，文档有没有给出可继续排查的方向？

    如果答案还不清楚，我会继续查资料或把不确定性写出来。起稿速度可以被工具提高，准确性仍然需要一条可回溯的证据链。
