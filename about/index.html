    关于我

    欢迎来到我的技术文档作品集。

    我是梁卓雯，正在面向技术文档工程师、资料开发工程师、开发者文档和技术内容岗位准备作品集。这个网站不是为了展示“我会做一个网页”，而是为了让招聘方看到：我如何理解技术资料、组织信息结构、设计阅读路径，并把文档纳入可维护的工作流。

    我有电子信息硕士背景，过去接触过英文论文、硬件规格书、接口说明、产品流程、课程资料和用户反馈整理。现在我把这些经历收束到一个方向：把复杂资料整理成别人查得到、读得懂、用得上的技术文档。

    我的工具箱

    Markdown 用结构化文本组织标题、表格、代码块、提示块和交叉链接。
    MkDocs Material 搭建可搜索、可导航、可发布的静态技术文档站点。
    Git / GitHub 用版本记录管理文档修改，让内容变化可追踪。
    GitHub Actions 把文档检查和发布流程接入自动化工作流。
    Vale 检查写作风格、术语一致性、格式和常见表达问题。
    Mermaid 把接口调用、SPI 时序和用户流程画成可维护的代码化图表。
    API Docs 组织 Endpoint、Header、参数、响应字段、错误码和调用示例。
    Hardware Datasheet Reading 拆解寄存器、通信协议、时序关系、配置步骤和代码示例。

    技术文档写作能力

    我目前重点展示四类技术文档能力：

    • 任务型文档：用 Quick Start、How-to Guide 和 Troubleshooting 帮助读者完成具体操作，并补充预期结果和排查路径。
    • 开发者文档：用 API 文档说明认证流程、Endpoint、请求参数、响应字段、错误码和调用示例。
    • 硬件资料重构：把硬件数据手册中的时序、寄存器和配置逻辑改写成更容易检索和复用的 Web 文档。
    • 产品文档：围绕小程序原型整理用户手册、PRD、版本日志和截图型帮助文档。

    对应样稿可以从 作品总览 进入；如果想看写作类型划分，可以看 写作样稿。

    推荐阅读路径

    如果你只有 5-8 分钟，我建议按这个顺序看：

    1. 文档质量自动化流水线：了解我如何把文档写作、检查、发布和维护串成系统；
    2. 快速开始：看任务型文档是否写得清楚、可执行；
    3. IoT 接口集成指南：看 API 文档结构和开发者视角；
    4. 硬件数据手册重构：看复杂技术资料如何转成结构化说明；
    5. 作品集改版复盘：看我如何判断信息架构问题并做取舍。

    如果你想先判断岗位匹配，可以直接看 简历。如果你想看持续思考和方法论，可以看 文稿 / 思考。

    中英文技术文档

    这个作品集以中文为主，因为当前目标更偏向国内技术文档、资料开发和技术内容岗位。英文能力不会用整站机翻来证明，而是通过更具体的证据呈现：

    • CET-6 612，具备英文技术资料阅读基础；
    • 研究生阶段阅读和分析过大量英文论文；
    • 作品集后续会优先补英文 Quick Start、英文项目概览和英文产品说明样稿；
    • 英文页面会作为精选样稿出现，不会替代中文主站。

    我如何处理一项文档任务

    我通常先确认读者是谁，以及读者要完成什么任务。然后把原始资料拆成概念、步骤、参数、示例、错误处理和下一步，用表格、代码块、流程图或截图承载最适合结构化的信息。

    我比较在意两个细节：第一，读者做完一步以后应该看到什么；第二，读者看完这一页以后应该去哪里。很多文档不是写得不够多，而是缺少预期结果和继续阅读路径。
