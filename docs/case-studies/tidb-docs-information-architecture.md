# TiDB 文档中心如何同时服务开发、部署与运维

> **案例性质**：公开文档复盘，不代表我参与过 PingCAP 或 TiDB 文档的编写与维护。本文基于 2026 年 7 月可访问页面进行分析。

**分析对象**：[TiDB 产品文档](https://docs.pingcap.com/zh/tidb/stable/)

大型产品文档最难处理的地方，往往不是缺少内容。开发者想找示例，运维人员关心升级、监控和故障，准备试用的人只想尽快跑起来。内容越完整，入口越容易变成一长串目录。

TiDB 文档中心同时保留了产品分类导航和任务入口。我打开首页时，可以从“快速上手、部署、迁移、运维、监控、调优”进入，也可以从“了解、试用、开发、工具、参考”找到当前角色更需要的内容。

## 首页先处理分流

这套首页没有试图解释所有概念。它把常见任务放到更靠前的位置，让读者先确认自己要做什么。

“开发”下面连接开发者手册、快速开始和示例程序；“运维”连接升级、扩容、备份恢复和巡检；“监控”与“调优”各自保留独立入口。不同角色共用一个站点，但不必先读同一套介绍。

## 版本是信息架构的一部分

页面明确显示当前文档版本，并保留发布历史。对于数据库产品，这一点直接影响配置、工具和操作步骤能否适用。

版本入口和任务导航同时存在，读者可以先确认“我用哪个版本”，再判断“我要做什么”。这两个问题处理清楚以后，大量内容才不会互相干扰。

## 大型文档站先做分流

整理复杂文档站时，我会先列出主要角色和高频任务，再检查首页是否真的提供了对应入口。内容类型也要分开：教程承担学习过程，操作指南服务具体任务，参考页负责稳定查找。

对于长期维护的产品，版本选择、发布历史、反馈和贡献入口同样属于阅读路径。它们决定读者是否敢把当前页面当成可靠依据。
