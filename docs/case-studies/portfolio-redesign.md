# 我如何把作品集从“简历堆叠”改成一个小型文档系统

## 项目背景

这个案例记录我对个人技术文档作品集的结构化改版。原站点已经有自动化文档流水线、硬件资料重构、API 文档和小程序文档，但内容更像“页面堆叠”：信息不少，证据链不够清楚。

改版目标不是做一个更花哨的网站，而是让招聘方能在 5-8 分钟内判断三件事：

* 我是否具备技术文档岗位需要的资料理解能力；
* 我是否能把复杂内容组织成可阅读、可维护的文档系统；
* 我是否能解释自己的文档结构和工具链选择。

## 技术选型 / 痛点

我保留 MkDocs Material，而不是迁移到更像个人博客的主题。原因是这个作品集本身要证明技术文档能力，MkDocs 的导航、搜索、侧边栏和 Markdown 工作流更接近真实文档站。

改版前主要痛点包括：

* **首页职责过重**：个人简介、项目、经历和技能混在一起，招聘方需要自己找重点；
* **导航层级不清**：Quick Start、Style Guide、Workflow 等页面没有被包装成完整文档集；
* **作品证据不够密集**：作品页缺少目标读者、交付形式和局部预览；
* **案例与作品混用**：最终交付物和过程复盘没有分开；
* **阅读路径断裂**：页面末尾缺少下一步入口。

## 执行过程

我先重排信息架构，再调整视觉和交互。核心思路是把网站拆成三个不同证据层：

```mermaid
flowchart LR
    A[首页：快速判断定位] --> B[关于：能力与工具箱]
    B --> C[作品：可交付样稿]
    C --> D[案例：过程复盘]
    D --> E[思考与经验：方法论解释]
    E --> F[简历：正式岗位匹配]
```

导航层面，我把“页面平铺”改成“入口 + 文档集”的关系：

```yaml
nav:
  - 首页: index.md
  - 关于:
      - 关于我: about.md
      - Changelog: site-changelog.md
  - 思考与经验:
      - 思考与经验总览: posts/index.md
  - 作品:
      - 作品总览: portfolio.md
      - 文档质量自动化流水线:
          - 项目概览: 01-automation.md
          - 快速开始: install.md
          - 写作风格指南: style-guide.md
          - GitHub Actions 工作流: github-actions-workflow.md
```

内容层面，我做了三类改动：

* 首页只保留定位、关键工具、主入口和版本信息；
* 作品页改成证据卡片，每张卡固定呈现 Context、缩略图和样稿入口；
* 简历页压缩为岗位匹配型结构，项目只保留背景、挑战、解决方案和产出。

## 项目反思

这次改版让我更明确地区分了三类内容：

* **作品** 证明我能交付什么；
* **案例** 证明我如何发现问题、选择方案并落地；
* **思考与经验** 证明我理解方法论，而不是只会堆页面。

我也意识到作品集本身就是一份技术文档。它需要读者路径、版本记录、链接检查、视觉一致性和长期维护机制。对技术文档岗位来说，这些细节比单纯的页面数量更能说明能力。
