# Docker Workshop 如何把概念说明接到可运行结果

> **案例性质**：公开文档复盘，不代表我参与过 Docker Docs 的编写或维护。本文基于 2026 年 7 月可访问页面进行分析。

**分析对象**：[Docker Workshop](https://docs.docker.com/get-started/workshop/)

容器、镜像、端口和 Compose 很容易被写成一组定义。读者看懂名词以后，仍可能不知道怎样把自己的应用真正跑起来。

Docker Workshop 给这段学习安排了约 45 分钟，并把目标写得很具体：构建并运行镜像、分享镜像、部署包含数据库的多容器应用，最后使用 Docker Compose。

## 概念只解释到足够开始

开头先说明容器和镜像。解释没有独立停留很久，下一步就进入一个简单的待办应用。读者随后会拉取代码、编写 Dockerfile、构建镜像并启动容器。

每个命令都连接一个可观察结果。容器启动以后，读者可以打开本地页面，也可以用 `docker ps` 检查运行状态。抽象概念在这里变成了文件、命令和页面。

## 后续章节沿用同一个应用

工作坊继续处理更新、分享、数据持久化、绑定挂载、多容器应用和 Compose。读者已经熟悉待办应用，因此可以把注意力放到当前新增的问题上。

每章结尾都有总结和下一步。教程没有在“运行成功”处突然结束，而是把第一次成功接到更完整的开发流程。

## 教程需要把读者送到下一个结果

写工具链教程时，我会让概念解释紧贴当前动作，并为关键步骤提供验证方法。一个连续样例也比多组互不相关的命令更容易积累理解。

如果教程较长，总时长、阶段目标和每章的下一步会直接影响读者是否愿意继续。这些信息本身就在帮助读者安排学习路径。
