# 写作风格指南

## 页面目标

本文档定义本项目采用的基础写作规范，用于提升技术文档的一致性、可读性和可维护性。

这些规则一部分来自写作判断，一部分可以交给 Vale 重复检查。两者一起构成这套文档的质量标准。

## 适用范围

本指南适用于本作品集中的技术文档页面，包括：

* Quick Start；
* Guide / How-to；
* API Reference；
* Troubleshooting；
* Changelog；
* 项目说明和案例复盘。

## 设计原则

本项目的写作规范遵循以下原则：

1. **清晰优先**：句子应直接、明确，避免不必要的修饰。
2. **任务导向**：操作文档应说明目标、步骤和预期结果。
3. **术语一致**：同一概念在不同页面中应使用统一表达。
4. **可验证**：关键步骤后应给出验证方式或预期输出。
5. **可维护**：规则应能被复用、检查和持续更新。

## 语气基调

同一个网站里，页面承担的任务不同，语气也不需要完全一样。

* **技术参考页**直接说明条件、动作、参数和结果。第一人称只在解释设计取舍时出现。
* **总览与案例页**可以从一次真实摩擦或观察开始，让读者看见问题怎样出现、判断怎样形成。
* **求职页面**先写事实和可打开的作品，再说明这些经历与岗位的关系。
* **不确定内容**明确标注模拟、原型、计划或待验证，不把推测润色成项目成果。

全站避免空泛口号和重复的对比句式。与其写“这个页面证明我具备某项能力”，更适合直接说明页面采用了什么结构、服务哪类读者，以及读者可以怎样检查结果。

## 规则 1：中英文之间保留空格

在中文技术文档中，中文与英文单词、数字之间应保留一个空格，以提升可读性。

| 正确示例                    | 错误示例                |
| ----------------------- | ------------------- |
| `这是一个 GitHub 仓库。`         | `这是一个GitHub仓库。`       |
| `今天的气温是 25 度。`            | `今天的气温是25度。`          |
| `使用 Vale 检查 Markdown 文件。` | `使用Vale检查Markdown文件。` |

### 可自动检查性

该规则可以通过 Vale 自定义正则规则进行检查。

示例规则：

```yaml
extends: existence
message: "中文与英文或数字之间应保留一个空格。"
level: error
scope: text
tokens:
  - '[\u4e00-\u9fa5][a-zA-Z0-9]'
  - '[a-zA-Z0-9][\u4e00-\u9fa5]'
```

## 规则 2：使用主动语态

技术文档应优先使用主动语态，让责任主体和操作动作更清楚。

| 推荐写法                 | 不推荐写法                  |
| -------------------- | ---------------------- |
| 浏览器发送请求。             | 请求被浏览器发送。              |
| 用户点击 Submit 按钮。      | Submit 按钮被用户点击。        |
| GitHub Actions 运行检查。 | 检查由 GitHub Actions 运行。 |

### 为什么这样写

主动语态更直接，读者更容易判断“谁执行了什么操作”。在操作型文档中，这可以降低理解成本。

## 规则 3：避免不必要的客套表达

技术文档应直接说明操作，不需要过度客套。

| 推荐写法                         | 不推荐写法                                        |
| ---------------------------- | -------------------------------------------- |
| Click the Submit button.     | Please click the Submit button.              |
| Run the following command.   | Please note that you should run the command. |
| Open the configuration file. | You may want to open the configuration file. |

### 为什么这样写

技术文档的重点是帮助读者完成任务。过多客套表达会增加句子长度，降低指令清晰度。

## 规则 4：操作步骤要包含预期结果

在 Quick Start 或 How-to 文档中，每个关键步骤应尽量说明预期结果。

| 推荐写法                           | 不推荐写法           |
| ------------------------------ | --------------- |
| 运行 `vale -v`。如果终端输出版本号，说明安装成功。 | 运行 `vale -v`。   |
| 打开 Markdown 文件。你应能看到 Vale 提示。  | 打开 Markdown 文件。 |
| 提交代码后，GitHub Actions 应自动开始检查。  | 提交代码。           |

### 为什么这样写

预期结果可以帮助读者判断自己是否完成了当前步骤，也方便后续排查问题。

## 规则 5：代码块应标明语言和用途

技术文档中的代码块应尽量标明语言、文件名或用途，避免读者不知道代码应放在哪里、如何执行。

| 推荐写法               | 不推荐写法          |
| ------------------ | -------------- |
| 使用 `bash` 标记命令行代码块 | 不标明代码块语言       |
| 使用 `yaml` 标记配置文件   | 把配置和命令混在同一个代码块 |
| 在正文中说明代码块属于哪个文件    | 只贴代码，不解释上下文    |

示例：

```yaml
# .vale.ini
StylesPath = styles
MinAlertLevel = suggestion

[*.md]
BasedOnStyles = Microsoft, MyStyle
```

## 规则 6：Troubleshooting 标题应接近真实问题

故障排查页面的小标题应尽量使用用户会搜索的问题或错误信息。

| 推荐标题                   | 不推荐标题 |
| ---------------------- | ----- |
| Vale Command Not Found | 安装问题  |
| VS Code Vale 插件没有提示    | 插件问题  |
| GitHub Actions 检查失败    | CI 问题 |

### 为什么这样写

真实错误信息更容易被搜索，也能帮助读者快速判断当前问题是否匹配。

## 与 Vale 的关系

本指南中的部分规则可以通过 Vale 自动检查，例如：

* 中英文之间缺少空格；
* 不推荐词汇；
* 过度客套表达；
* 术语不一致；
* 特定格式问题。

并不是所有写作质量都能完全自动化检查。Vale 更适合处理可规则化、可重复识别的问题；对于逻辑结构、信息完整性和用户路径，仍需要人工审查。

当新增文档类型、术语或自动检查规则时，应同步更新：

* 写作风格指南；
* Vale 规则；
* Troubleshooting；
* Changelog。
