# 文档为什么会过期：我如何给内容留下维护线索

文档过期时，页面通常仍然打开，句子也依然通顺。

真正的问题藏在细节里：界面按钮换了名称，默认超时时间已经修改，响应多了一个字段，原来的截图还在告诉读者点击旧入口。

这种错误很难靠一次全面检查彻底解决。更可靠的做法，是让变化能够留下线索。

## 我会先找内容和什么绑定

每一页文档都依赖某些外部事实。

API 页面依赖接口定义和版本；用户手册依赖界面和操作流程；硬件资料依赖器件规格；安装指南依赖操作系统、工具版本和命令。

如果这些依赖没有被记录，产品变化后很难知道哪些页面需要一起更新。

我会在项目中建立最基本的对应关系：功能或接口变化时，哪些文档属于检查范围；一个配置值来自哪份规格；截图对应哪个版本和操作路径。

## 把维护信号放进工作流

文档维护可以借助几类信号：

* 代码或配置变化触发相关文档检查；
* Pull Request 模板询问是否需要更新文档；
* 链接检查发现失效入口；
* Changelog 记录新增、修改和移除内容；
* 页面标注适用版本或最后核对时间；
* 读者反馈进入可追踪的问题列表。

这些机制无法保证内容永远正确，但能减少“变化发生了，文档完全没有收到消息”的情况。

## 自动化适合检查确定规则

在[文档质量自动化流水线](../01-automation.md)里，我使用 Vale 和 GitHub Actions 检查格式、术语和构建流程。这类规则清楚、可重复，适合自动执行。

技术事实是否仍然有效，需要新的来源或实际测试。自动化可以提醒链接失效，却无法判断页面描述的业务流程是否已经改变。维护工作需要同时保留机器检查和人工复核。

## 更新记录应该帮助读者判断影响

Changelog 如果只写“优化文档”“修复问题”，读者很难知道是否与自己有关。

我更倾向于说明变化对象和读者影响，例如：

* 更新本地 Vale 安装命令，适配新的包名称；
* 补充 `401` 错误的 Token 排查步骤；
* 移除旧版界面截图，并更新按钮名称；
* 为寄存器表增加复位值说明。

这样的记录既是发布信息，也是下一次维护时的线索。

## 我目前能做到的范围

这个作品集规模不大，我可以通过构建、链接检查和人工浏览维护它。它还没有真实产品仓库、发布节奏和支持工单，因此不能模拟完整企业文档治理。

但我可以先建立习惯：记录来源、标注模拟内容、保留更新记录、让核心页面互相链接，并在改动后重新走查读者路径。

文档会过期是一件正常的事。维护能力体现在变化发生以后，我们能否知道该检查哪里，以及如何让读者看见变化。
