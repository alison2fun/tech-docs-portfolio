# 同一份技术资料，为什么需要不同阅读路径

一份硬件规格可以同时被嵌入式开发者、测试人员和技术支持使用。

他们面对的是同一颗器件，却不会从同一页开始，也不会在同一个答案上停下。把所有信息放进一份长文档，只能保证内容存在，无法保证每个人都能找到自己的路径。

## 先区分读者要完成的任务

我会先写下不同读者进入文档时的问题。

嵌入式开发者可能要完成初始化和数据读取；测试人员需要确认配置、边界和预期输出；技术支持更关心常见故障、日志和恢复步骤；产品协作者则需要了解功能范围和限制。

这些任务会共享一些事实，例如寄存器定义或接口字段，但使用方式不同。

## 共享事实只维护一份

重复复制参数表和字段说明，很容易产生版本差异。

我倾向于把稳定事实放在 Reference 中，再从任务页面链接过去。Quick Start 只保留第一次成功所需的信息，Troubleshooting 按症状组织，概念说明解释系统如何工作。

这样既能让读者沿任务前进，也能避免同一个字段在多个页面被重复维护。

## 页面开头要帮助读者判断是否适用

标题下面的一小段说明很重要。它可以告诉读者：

* 这页帮助完成什么；
* 适用于哪个产品或版本；
* 开始前需要知道什么；
* 如果目标不同，应该去哪里。

这几句话能减少读者读到一半才发现页面不适用的情况。

## 导航需要表达内容分工

当栏目名称都是“概述”“更多信息”“高级配置”时，读者很难预测页面内容。

我会优先使用任务或对象命名，例如“获取设备实时数据”“配置 SPI 通信”“排查 Token 认证失败”。名称越接近读者的问题，导航越能承担寻找答案的工作。

## 作品集也使用同样的方法

这个网站面对的读者包括 HR、用人经理和技术文档负责人。他们共享“判断候选人是否匹配”的目标，关注点仍有差异。

[关于我](../about.md)提供岗位方向和快速路径，[作品总览](../portfolio.md)展示交付物，[案例复盘](../case-studies/index.md)记录分析与取舍，[简历](../简历.md)提供事实依据。页面分工让读者可以从自己的问题出发。

设计多条路径不意味着维护多套互相重复的内容。更稳妥的做法，是让事实集中、任务分流、页面之间建立清楚链接。读者可以走不同路线，最终看到的证据仍然一致。
