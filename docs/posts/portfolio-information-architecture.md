# 技术文档作品集为什么需要信息架构

<div class="post-meta">思考与经验 · 信息架构 · 2026-07</div>

在搭建这个作品集时，我有一段时间总是不太满意。

最初的版本内容也很多。我把做过的文档放上去，加上简历，补充项目说明，告诉读者我做过什么。

问题出在阅读过程上。信息已经摆出来了，读者仍然需要自己判断从哪里开始、哪些页面值得看、几份材料合起来能说明什么。

招聘方打开一个陌生候选人的网站，通常只想尽快得到四个答案：这个人想找什么岗位、有没有相关能力、有哪些作品可以证明、如果感兴趣下一步该点哪里。

“从一堆页面里拼出答案”的形式不适用于作品集，也不适用于技术文档。

我希望能用一种更清楚的方式，让读者拿到自己需要的信息。

## 作品集也是一份技术文档

于是，我开始重新把这个网站当成一份技术文档来处理。

在我的理解里，一份好的技术文档需要入口、导航和阅读路径，也需要让读者知道每一部分内容解决什么问题。作品集同样要处理这些事情。

它的第一读者是招聘方，包括 HR、用人经理，以及可能参与筛选的技术文档负责人。他们更可能先快速扫一遍，再决定要不要深入阅读。所以我需要主动说明岗位方向、能力证据和继续阅读的位置。

具体来说，我从读者的目标出发，把信息架构拆成四个部分：

- **入口**：读者从哪里开始看？
- **路径**：读者接下来应该看什么？
- **分工**：每个页面负责证明什么？
- **证据链**：这些页面合起来证明什么能力？

按照这四个维度，我给网站里的页面重新分工。

**[关于我](../about.md)** 负责说明我的目标岗位、专业背景和核心能力，并给第一次访问的人一条 5 分钟阅读路径。

**[作品总览](../portfolio.md)** 负责展示可交付成果。每张卡片都会说明项目性质、目标读者、解决的问题、交付物和主要证据，模拟项目也会明确标注。

**[写作样稿](../writing-samples/index.md)** 按文档类型整理现有页面。招聘方可以直接查看 Quick Start、Workflow、Troubleshooting、API Reference、用户手册和 PRD。

**[案例复盘](../case-studies/index.md)** 记录发现问题、做取舍、执行修改和验证结果的过程。它保留的是我如何判断和行动。

**[思考与经验](index.md)** 收集方法论和学习心得，记录我如何理解技术文档工作。

**[简历](../简历.md)** 提供教育、经历、语言和工具等事实，方便招聘方对应岗位要求。

现在的阅读路径是：从 [首页](../index.md)确认岗位方向，进入 [关于我](../about.md)了解背景，再根据时间选择 [作品总览](../portfolio.md)或[写作样稿](../writing-samples/index.md)。想看过程，可以继续到[案例复盘](../case-studies/index.md)；想看方法论，可以进入[思考与经验](index.md)；需要核对经历和联系方式时，再打开[简历](../简历.md)。

## 用“5 分钟标准”来验证这套架构

针对求职场景，我给这个作品集设置了一个很朴素的标准。

如果招聘方只花 5 分钟浏览网站，TA 是否能得到这些信息：

1. 我想找的岗位是什么。
2. 我有什么技术文档相关能力。
3. 哪些作品可以证明这些能力。
4. 如果想继续了解，下一步应该点哪里。

任何一个答案模糊，都说明对应的入口或路径还需要调整。

我用这个标准反复走查网站，也请几位有招聘或用人经验的朋友做过盲测：不给背景说明，让他们直接打开首页，5 分钟后再回答这四个问题。

前两轮测试里，第 3 和第 4 个问题的回答不够顺畅。它们直接推动了后来的修改：作品卡增加项目性质、目标读者和主要证据；首页重新安排按钮权重；简历提供 PDF 入口；核心页面补上底部导航。每一次调整都对应一个实际暴露出来的阅读问题。

## 内容本身就是一次交付

在我的理解里，对技术文档工程师来说，作品集本身就是一次文档交付。

它需要判断读者是谁，读者要完成什么任务，哪些信息应该优先出现，哪些内容可以放到后面，页面之间又该如何连接。这些判断，和整理一份 API 文档或一本数据手册时需要做的工作是一致的。

这篇文章也要承担样稿的作用。读者应该能看出问题从哪里出现、我做了哪些取舍、目前的网站如何回应这些问题。能否做到这一点，比写出一个漂亮的概念更重要。

这就是我希望这个作品集完成的事：内容本身即是交付，阅读过程即是验收。
