# 选择文档工具前，我会先问哪几个问题

文档工具很容易变成一场功能比较。

支持多少主题、能不能在线协作、有没有 AI、导出格式是否丰富，这些信息都值得看。但真正开始维护以后，最常影响体验的往往是更普通的问题：谁来写、多久更新、如何评审、发布失败时谁能处理。

所以我在选择工具前，会先回答下面几个问题。

## 读者在哪里完成任务

读者需要在产品界面旁查看帮助，还是在开发时搜索 API？他们使用手机、桌面浏览器，还是离线 PDF？

阅读场景决定交付形式。开发者文档通常需要稳定链接、代码块、搜索和版本说明；设备现场资料可能更看重离线访问和打印；内部流程文档则需要权限和协作记录。

工具只有进入真实场景以后，功能才有优先级。

## 作者和评审者如何协作

如果内容主要由技术写作者维护，结构化文本和版本控制可能很合适。如果产品、支持和销售需要频繁直接编辑，可视化编辑和评论流程会更重要。

我还会检查评审发生在哪里。代码评审团队可以在 Pull Request 中同时审代码和文档；习惯在线文档的团队则可能更依赖评论、修订记录和权限控制。

选择一种工具，也是在选择协作摩擦出现的位置。

## 内容多久变化一次

更新频率会影响发布方式。

一年只更新几次的固定手册，可以接受较重的排版和人工发布。跟随版本持续变化的开发者文档，需要更短的反馈链：修改、检查、预览和发布最好能重复执行。

如果文档和产品版本并行存在，还要提前确认版本切换、旧版保留和失效内容如何处理。

## 哪些质量规则可以被检查

格式、链接、术语和构建错误可以通过工具检查多少？哪些内容仍需要人工评审？

我不期待一套工具判断文档是否真正有效，但希望它能稳定处理机械问题，让评审时间留给技术准确性和读者路径。

这也是我在当前作品集中使用 Vale、MkDocs 和 GitHub Actions 的原因。它们分别处理写作规则、站点构建和提交后的自动检查。

## 团队能否长期维护

最丰富的工具如果只有一个人会配置，维护风险仍然很高。

我会查看安装成本、插件依赖、升级方式、导出能力和迁移难度。还会问一个很实际的问题：半年后换人接手，现有内容和构建流程是否容易理解？

当前作品集选择 Markdown 和 MkDocs，是因为它们适合公开样稿、代码示例和 Git 工作流。这个选择服务于我的内容规模和求职展示，并不意味着它适合所有团队。

工具评估最后应该落到一组约束：读者任务、作者能力、评审方式、更新频率、质量检查和维护成本。把这些条件写出来，比单独比较功能列表更容易得到可解释的选择。
