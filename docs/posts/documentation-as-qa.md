# 为什么写文档会提前暴露产品问题

写文档时，我经常会遇到一种停顿：这一步到底发生在保存之前，还是保存之后？失败以后页面会保留输入吗？同一个字段为什么在两处使用了不同名称？

这些问题表面上影响文字，继续追下去却可能指向产品流程、接口定义或交互状态里的空白。

## 文档会逼迫流程变得完整

开发和设计可以暂时依赖团队内部的共同理解。文档要把这个理解交给一个不在现场的读者，因此每一步都需要说清楚。

当我尝试写一条完整任务时，会自然检查：

* 操作从什么状态开始；
* 用户需要哪些权限和数据；
* 每一步会改变什么；
* 成功后出现什么结果；
* 失败时系统如何反馈；
* 下一步可以继续做什么。

如果其中一项没有答案，文档就会出现模糊词，例如“配置完成后”“按需要设置”或“系统将自动处理”。这些词提醒我，流程可能还没有被真正说明。

## 命名问题往往来自边界问题

同一个对象在界面、接口和内部讨论中使用不同名称，会直接增加文档成本。

写作者必须决定采用哪个词，还要解释其他名称是否指向同一对象。这个过程有时会发现：团队对概念边界本来就没有一致答案。

术语表可以记录最终决定，Vale 可以检查已知术语是否被一致使用。但工具无法替团队决定两个概念是否应该合并。这个判断需要产品、开发、测试和文档共同确认。

## 示例会检验接口是否自洽

API 文档尤其容易暴露问题。

当请求示例、参数表和响应示例放在同一页时，字段缺失、类型冲突和错误码空白会变得明显。代码能运行却无法解释，并不代表接口对使用者足够清楚。

在我的模拟 [IoT 接口文档](../03-api.md)里，我会逐项对照 Endpoint、设备 ID、认证 Header 和响应字段。由于它没有真实服务端，我只能验证文档内部一致性，不能声称完成了真实接口测试。这个限制也应该写在作品说明里。

## 文档问题和产品问题需要分开记录

发现疑点以后，我会先判断它属于哪一类：

* 页面缺少说明或链接，属于文档问题；
* 规格已经明确，但界面或接口行为不一致，可能是实现问题；
* 多份来源互相冲突，需要负责人确认；
* 当前版本尚未定义，需要记录为开放问题。

清楚分类可以避免文档自行“修复”产品。写作者可以指出缺口、提供复现步骤和说明读者影响，最终定义仍应由相应负责人确认。

对我来说，文档的 QA 价值来自这种逐步追问。它让隐含知识变成可以检查的句子、步骤和示例，也让尚未解决的问题更早出现。
