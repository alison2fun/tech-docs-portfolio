    我是谁¶

    我是梁卓雯，正在准备技术文档工程师、资料开发工程师、技术内容 / 海外技术内容相关岗位。

    文档不是把信息放上去就完事。

    它要有入口，有路径，有下一步，有排错，有维护记录。读者看完一页之后，最好知道自己接下来该去哪。

    这也是我现在想通过这个作品集展示的能力。

    我想展示的 3 类能力¶

    1. 技术文档写作能力¶

    我能把一个技术流程拆成清楚、可执行、能验证的文档。

    这件事听起来很普通，但真的写起来会发现，很多文档卡住读者的地方，不是信息不够，而是没有告诉他每一步做完应该看到什么。

    所以我在 Quick Start 里补了适用读者、前置条件、操作步骤、预期结果和常见问题。

    相关页面：

    • 快速开始，搭建本地文档质量检查环境
    • 写作风格指南，沉淀写作规则、示例和术语表
    • 故障排查，用“现象、可能原因、解决方法、验证方式”组织排错内容
    • 任务拆解小程序用户手册，展示面向用户的功能说明
    • API 参考文档，展示接口说明和结构化表达能力

    2. 文档工程化意识¶

    我越来越觉得，技术文档很像一个长期运行的小系统。

    它会新增页面，会改术语，会出现旧链接，会有写作规则不统一的问题，也会有本地检查通过、线上检查失败这种很真实的麻烦。

    所以我做了一个“文档质量自动化流水线”作为核心案例，把 Markdown、Vale、GitHub Actions 和 MkDocs 串起来，模拟一个轻量级 Docs-as-Code 工作流。

    相关页面：

    • 文档质量自动化流水线，查看项目背景、工具链和整体工作流
    • GitHub Actions 工作流，查看提交后的自动化检查流程
    • 更新记录，查看这个文档项目的结构和内容如何迭代
    • 页面之间的内部链接和推荐阅读路径，用来减少读者迷路

    这件事对我来说很重要。

    它让我不只是停留在“我写了一些文档”，而是开始考虑这些文档怎么被检查、怎么被维护、怎么被别人顺着读下去。

    3. 中英文技术内容表达能力¶

    我希望未来能做技术文档、资料开发，也想争取一些海外技术内容相关的机会。

    英文这块我现在不打算一口气把全站翻完。那样很容易变成机械翻译，工作量也很大。

    我会先做最小可展示版本，优先处理三个页面：

    • Start Here
    • 文档质量自动化流水线项目概览
    • Quick Start

    这三页最能说明问题。一个负责职业定位，一个负责项目解释，一个负责任务型技术文档。

    后续我会继续补充中英文技术写作样稿，包括产品说明、教程、API 说明和面向海外读者的技术内容。

    推荐阅读路径¶

    如果你是面试官，可以按这个顺序看：

    1. 文档质量自动化流水线 先看我的核心文档工程化案例，了解这个项目解决什么问题。

    2. 快速开始 看我如何写任务型文档，让读者完成一次本地文档质量检查。

    3. 写作风格指南 看我如何把写作规范整理成可复用、可检查的规则。

    4. GitHub Actions 工作流 看我如何把文档质量检查接入自动化流程。

    5. 作品集改版复盘 看我如何把这个网站从“简历堆叠”改成一个小型文档系统。

    如果你想看完整经历，可以查看 简历。

    这个作品集现在怎么读¶

    • 文档作品，放正式技术文档，包括 Quick Start、Guide、API Reference、Troubleshooting 和 Changelog。
    • 案例复盘，记录我怎么发现问题、怎么调整结构、怎么复盘决策。
    • 写作样稿，后续会放中英文技术写作、产品说明和教程样稿。
    • 简历，放完整教育背景、项目经历、技能和联系方式。

    我的文档理念¶

    我现在对技术文档的理解很简单。

    读者要知道从哪里开始。 读者要知道下一步去哪。 读者出错的时候，要能找到排查办法。 文档改过什么，也应该留下记录。

    这几件事做到了，一个文档站才会真的像一个系统。

    这个作品集目前还在迭代，但它已经开始变成我想要的样子，一个能展示技术写作、信息架构、文档工程化和持续维护意识的小型文档产品。
