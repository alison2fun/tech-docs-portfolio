    技术文档作品集

    面向技术文档工程师 / 资料开发工程师 / 技术内容岗位

    我把这个网站当成一个小型文档产品来改。从首页入口、导航结构，到 Quick Start、故障排查和更新记录，我一步步把原来比较散的内容整理成一个可阅读、可维护的文档系统。

    这件事让我更清楚地理解：文档要有入口、有路径、有下一步，也要有维护记录。我希望面试官看到的，不是一个静态页面，而是我如何把技术内容一点点搭成文档系统。

    01 Infrastructure

    文档质量自动化流水线

    我把 Markdown、Vale、GitHub Actions 和 MkDocs 串起来，做了一个轻量级 Docs-as-Code 案例。这个项目记录了文档如何被编写、检查、发布和持续维护，也让我开始真正理解“文档工程化”不是工具堆叠，而是一套能长期运行的流程。

    项目概览 → GitHub Actions 工作流 → 更新记录 →
    02 Writing

    技术文档写作能力

    我在这个作品集里补了 Quick Start、写作风格指南、故障排查和 API 参考文档。写这些页面的时候，我发现很多文档卡住读者的地方，往往出在操作路径和预期结果没有写清楚。所以我会在任务型文档里补适用读者、前置条件、操作步骤、预期结果和常见问题。

    快速开始 → 写作风格指南 → 故障排查 →
    03 Reading Path

    推荐阅读路径

    如果你是面试官，可以按这个顺序看：

    • 文档质量自动化流水线 → 了解核心项目解决什么问题
    • 快速开始 → 看我如何写任务型文档
    • 写作风格指南 → 看我如何沉淀写作规则
    • GitHub Actions 工作流 → 看我如何把文档检查接入自动化流程
    • 作品集改版复盘 → 看我如何把网站从简历堆叠改成文档系统
    04 Bilingual

    中英文技术内容表达能力

    英文部分我不会一口气翻完整站。第一阶段我会先做三个页面：Start Here、项目概览、Quick Start。这三个页面分别对应英文职业定位、英文项目解释和英文任务型技术文档。先把最关键的页面打磨好，再继续扩展英文样稿。

    英文 Start Here（计划中） → 英文项目概览（计划中） → 英文 Quick Start（计划中） →

    读者要知道从哪里开始，下一步去哪，出错时能找到排查方法，文档改过什么也要留下记录。

    这个作品集还在迭代中。我会继续把新的文档练习、项目复盘和英文样稿补进来，让它越来越接近一个真实的技术文档系统。
