    我如何把作品集从“简历堆叠”改成一个小型文档系统

    背景

    我之前已经有一个技术文档作品集网站。

    重新打开它的时候，我第一反应是：内容其实不少。首页有经历，项目页有自动化文档流水线、写作规范、快速开始、接口文档，小程序文档也放进去了。

    但看了一会儿，我有点不踏实。

    这些内容都在，可是它们没有被好好组织起来。面试官点进来，可能会看到很多信息，却不一定能快速判断：我到底想证明什么能力？应该先看哪个项目？这些页面之间有什么关系？

    这个发现挺关键的。

    我之前以为作品集主要是“把东西放上去”。这次改版让我意识到，作品集本身也需要信息架构。它应该像一个文档产品一样，有入口、有路径、有主案例，也有后续维护的痕迹。

    所以我开始重新整理这个网站。

    原来的问题

    我主要发现了 4 个问题。

    1. 首页承担了太多任务

    原来的首页像一份很长的简历。

    目标岗位、个人优势、项目经历、工作经历、技能、教育背景都放在一起。信息很全，但第一眼不够有重点。

    面试官进入网站后，其实最需要先知道三件事：

    • 我想应聘什么岗位；
    • 我最想证明哪几类能力；
    • 应该按什么顺序阅读这个作品集。

    这些问题没有被首页直接回答，所以内容再多，判断成本还是高。

    2. 顶部导航层级不清楚

    原来的顶部导航里有“项目案例”“写作规范”“快速开始”。

    我后来才意识到，“写作规范”和“快速开始”不适合作为全站一级入口。它们更像“文档质量自动化流水线”这个项目下面的具体页面。

    这有点像产品文档。一个安装步骤、一个风格规范，应该归在某个文档集下面。直接放到顶部导航，会让网站看起来像页面平铺，读者也更难理解这些内容之间的关系。

    所以我把它们下沉到了核心项目里。

    3. 核心项目没有被包装成完整案例

    “文档质量自动化流水线”是目前最能代表我文档工程化意识的项目。

    它里面有 Markdown、Vale、GitHub Actions、MkDocs，也有写作规范、故障排查和更新记录。单看每个页面都能说明一点能力，但原来它们散在不同入口里，整体感不强。

    我想让面试官看到：这不是几篇零散页面，这是一个围绕文档质量检查搭出来的小型 Docs-as-Code 案例。

    于是我把它重组为一个完整文档集。

    4. 页面之间缺少继续阅读路径

    原来的页面写完就结束了。

    读者看完 Quick Start，不一定知道接下来该看写作规范，还是看自动化检查。看完故障排查，也不一定能回到项目概览或更新记录。

    这个问题很细，但挺影响体验。

    技术文档里，页面之间要能互相接住。读者看完一页，最好知道下一步去哪。于是我开始给核心页面补“下一步阅读”，并把它们改成可点击的内部链接。

    我的改版决策

    这次改版，我先动结构，没有急着改样式。

    我把网站重构成 5 个一级入口：

    • Start Here：让面试官快速理解我的目标岗位、核心能力和阅读路径；
    • 文档作品：放正式技术文档作品；
    • 案例复盘：记录项目背景、问题、决策和反思；
    • 写作样稿：后续放中英文技术写作、产品说明和教程样稿；
    • 简历：保留完整经历，但不再让首页承载全部信息。

    这个调整之后，首页的职责变清楚了。

    它不再负责展示所有经历，只负责做一件事：帮第一次进入网站的人快速判断，我是谁、我在准备什么岗位、我的能力证据在哪里。

    核心案例：文档质量自动化流水线

    我把“文档质量自动化流水线”整理成一个完整文档集：

    • 项目概览
    • 快速开始
    • 写作风格指南
    • GitHub Actions 工作流
    • 故障排查
    • 更新记录

    这套结构对应的是一条比较完整的读者路径：

    1. 先理解这个项目解决什么问题；
    2. 再通过 Quick Start 跑通一个最小流程；
    3. 接着查看写作规则；
    4. 然后理解自动化检查怎么接入提交过程；
    5. 遇到问题时查看故障排查；
    6. 最后通过更新记录看到文档如何迭代。

    这个结构搭完之后，我对“技术文档系统”这件事更有感觉了。

    技术文档不只是写几篇页面。它要考虑读者从哪里进来，在哪一步会卡住，下一步需要什么，以及这些内容以后怎么维护。

    我学到的东西

    这次改版让我确认了几件事。

    第一，作品集也需要信息架构。 内容多只是基础。更重要的是，面试官能不能快速看懂这些内容在证明什么。

    第二，导航会影响别人怎么理解你。 Navbar 负责全站入口，sidebar 负责某个文档集内部的阅读路径。这个区分看起来很小，实际改完之后，网站立刻清楚很多。

    第三，技术文档要有“下一步”。 一个页面写得再完整，如果读者看完后不知道继续去哪，整个文档系统还是断的。

    第四，工具要服务文档问题。 Vale、GitHub Actions、MkDocs 都只是工具。真正重要的是，我能不能用它们解决文档质量、一致性、维护成本和发布流程的问题。

    改版结果

    这次改版后，作品集从“简历 + 项目页面”的形态，变成了一个更清楚的小型文档系统。

    现在它可以展示：

    • 我如何组织技术内容；
    • 我如何设计文档阅读路径；
    • 我如何把写作规范转化成可检查规则；
    • 我如何用 GitHub Actions 接入文档质量检查；
    • 我如何通过 Changelog 和 Troubleshooting 体现长期维护意识。

    这次改版本身也变成了一个案例。

    我没有只停留在“我理解文档工程化”这句话上，而是把这个理解用在了自己的作品集里。这个过程对我很有价值：一边改网站，一边把技术文档岗位真正需要的能力拆得更清楚。

    下一步计划

    接下来我会继续做几件事：

    • 继续补充真实的故障排查案例；
    • 优化 GitHub Actions 工作流说明；
    • 增加更多可点击的内部链接；
    • 为 Start Here、项目概览和 Quick Start 制作最小英文版；
    • 根据不同岗位继续调整作品集重点。

    这次改版让我更加确定了：

    作品集不是一次性做完的展示页。它更像一个可以持续更新的文档产品，会随着我的项目、理解和求职方向继续生长。
